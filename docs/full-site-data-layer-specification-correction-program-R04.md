## Program outcome

The correction is complete only when an operator can create, prove, publish, and operate a full Retail and Trade data-layer specification without source, persisted-state, stable-ID, or runtime-module access. The standalone Builder and side panel are complementary entrances to the same workflow rather than separate tools: saved schemas, captured validation, rich specification tables, authoring, compilation, Fixtures, coverage, release, and Live must expose one canonical executable truth.

The terminal boundary is the built installed extension at its `chrome-extension://` pages with a fresh browser profile and a real permitted observation target. Direct storage population, raw-ID form input, preconstructed rich entities, replaced Chrome APIs, direct production-callback invocation, cached cross-scenario outcomes, legacy validation as oracle, and component-only layout assertions cannot satisfy terminal scenarios.

## Canonical product decisions

- One revisioned `SpecificationProject` draft is the only mutable authority for Profiles, Pages, Events, Applicability Sets, Flows, steps, transitions, Fixtures, schemas, Assignments, evidence, and release metadata.
- Every mutation is a command-scoped patch with project and entity base revisions. Disjoint stale commands may merge deterministically; overlapping fields require visible resolution. Whole schema collections are never written from a surface snapshot.
- Compatibility data is migrated once or remains explicitly read-only. It cannot contribute rendered counts, compiler candidates, coverage, release content, or Live behavior through a parallel path.
- The side panel owns in-page capture, Live validation, and compact contextual editing; the standalone Builder owns project composition, broad rich authoring, assurance, and release. These are information-architecture roles, not capability or data silos. Each surface offers named Continue/Open actions into the other with project, environment, entity, field, draft or release, base revision, and return context.
- Adding a saved library schema to a project is a staged canonical adoption, not manual recreation or a hidden parallel link. The project receives an equivalent project-owned draft with source revision lineage; later library revisions are opt-in semantic diffs applied through reviewed synchronization while project-local overrides and published releases remain stable.
- Captured Live validation can continue directly into a named project as a guided Fixture or staged Profile requirements. The observation, evaluator result, proposed assertions, provenance, and destination context transfer without clipboard, export/import, raw IDs, or duplicate legacy records.
- The production rich specification-table builder is shared by side panel and standalone Builder. Hierarchical selection, effective origins, full seven-column derivation, completeness, column order, heading choice, table style, example overrides, Spreadsheet output, Rich table output, and safe clipboard fallback cannot be replaced by a reduced standalone table model.
- Human selectors display names and persist stable references. Renaming changes labels, not identity or runtime matching.
- Common Pages and Events have no Applicability by default. Scoping occurs only when the operator explicitly selects it.
- One compiled executable plan contains matchers, flow automata, ordered Profiles, effective schemas, Assignments, Fixtures, and provenance. One evaluation result contains candidates and rejection reasons, winner or tie, active Flow/step, Profiles, schema revision, issues, provenance, and state transition.
- Live evaluates the selected immutable published plan. Drafts affect Preview only until a new release passes the same gate and is published.
- Validation enforces nested and scalar types, allowed values, forbidden and required fields, cardinality, and every authored rule the interface offers. Every issue includes path, stable code, severity, expected, actual, and provenance.
- Canonical authoring preserves every rule capability currently available in the side panel: Required, Exact value, Allowed values, Regular expression, Text length, Digits only, Numeric range, Item count, Allow undeclared properties, reusable-rule lifecycle, severity, custom issue message, enabled state, version policy, and conditional parameters. Adoption or migration cannot silently downgrade one of these capabilities.
- The canonical condition/query model is a recursively typed All/Any/Not tree shared by Applicability, Flow entry/exit/transition conditions, and conditional validation rules. Its leaves preserve Exists, Does not exist, Equals, Does not equal, Is one of, Matches pattern, Is greater than, Is at least, Is less than, and Is at most, with typed literal, set, pattern, or compatible property-reference operands. Guided controls and a plain-language preview author the tree; free-form JavaScript and saved inert expressions are prohibited. An extensible operator may appear only after compiler, matcher, Fixture, preflight, release, Live, issue, provenance, and accessibility behavior exist for it.
- Release-gating Fixtures require observations and assertions. Required coverage cells require current proving Fixture assertions; waivers cannot manufacture evidence or make a zero-evidence project ready.
- Review and publication consume the same content-addressed preflight result. Any canonical change makes it stale.
- Live permission onboarding requests the current target origin first. Broad all-tabs access remains a separately explained expert option with explicit scope, persistence, revocation, and consequence text; it is never a hidden prerequisite for the terminal workflow.
- Blind human or agent interaction exercises are outside this correction cycle and do not gate recommendation completion. The owner may perform a separate Windows walkthrough later without converting it into required SwarmForge evidence.

`Add saved schema to project` stages an equivalent project-owned draft pinned to the selected library revision and records lineage. `Synchronize` is a separate reviewed command for a later library revision; it preserves local overrides and invalidates affected evidence. `Open` navigates to an already canonical entity and never creates a copy.

## Existing-specification reconciliation

The R04 additions retain stable scenario numbering and strengthen the named production boundary rather than duplicating broad older scenarios. In particular:

- Canonical draft and assignment scenarios now require visible state, canonical collections, and compiler input to be identical, including both stale-edit orders.
- Effective-schema and container/child scenarios now cover normal parent/child compilation order and complete issue structures rather than only generated document shape.
- Effective-schema scenarios preserve the complete existing side-panel rule catalog and lifecycle while adding nested typed query composition at the canonical compiler and installed-runtime boundaries.
- Canonical adoption scenarios now connect saved library schemas and captured validation to active projects with explicit lineage, synchronization, and cross-surface return context.
- Existing schema specification-builder and customization scenarios now require the same rich production component and configuration on both side panel and standalone surfaces.
- Applicability and Flow scenarios now require name selectors to persist stable references, common entities to remain unscoped, and editor transitions to be the runtime transitions.
- Fixture, coverage, and release scenarios now make empty proof, zero cells, ambiguity, compilation failure, validation failure, and mismatched gates decisive blockers.
- Unified evaluation and terminal runtime scenarios now require the installed observer and visible Live result from an immutable release; routing records and legacy validation are insufficient.
- Operator scenarios now cover concepts, sequence, action impact, repair, toolbar/inspector scope, narrow layout, exact focus, and a worked example. They do not require a blind role-play exercise.

Existing mutation stamps and manifests are historical tool output. They are not edited by this specification and become untrusted when their feature content hash changes.

## Evidence and acceptance ownership

- Behavior features describe external outcomes; paired runtime features must execute the named installed-extension or production boundary independently by scenario identity.
- Terminal greenfield setup begins from a clean browser profile and rendered blank-project path. It may not use project JSON, raw stable IDs, serializer calls, direct storage writes, or seeded rich entities.
- Live proof uses an explicitly permitted real safe page, page-emitted events, the extension observer, Chrome runtime messaging, and rendered Live results. Direct callback adapters are supporting tests only.
- Browser storage, focus, scrolling, geometry, permission prompts, release bytes, and displayed evaluation objects are asserted at their actual boundaries.
- Cross-surface evidence begins from the same named canonical source revision and compares destination context, semantic content, configuration, command revision, and output. Merely observing records in shared storage is insufficient.
- Each critical regression has its own negative production mutation or fault case. Failure in another scenario does not count as detecting it.
- No blind interaction run by human or Codex agents is required, scheduled, or represented as acceptance evidence. Any later Windows usability walkthrough is owned and interpreted separately by the user and does not reopen or block this recommendation cycle.
- R05 post-implementation evidence must preserve every R01–R04 input unchanged and record build identity, extension URL, clean-profile provenance, viewport, ordered actions, visible results, persisted observations, and release status.

## Assumptions and settled product decisions

### Settled assumptions

1. Disjoint stale commands auto-merge only when their affected fields and entity relationships do not overlap; overlap always opens visible resolution.
2. Browsing permission onboarding requests the narrowest target-origin access first. Persistent all-tabs access is a separately explained opt-in, not an implicit prerequisite.
3. Every matcher operator shown in the editor must execute in the production matcher; unsupported operators are absent rather than saved as inert configuration.
4. Required release coverage needs current assertion-bearing Fixture evidence. A waiver may document an optional gap but cannot satisfy a required cell or turn zero evidence into readiness.
5. Flow-instance correlation retains the approved browser-tab-session default with an optional explicit transaction/session key; equal active instances remain an ambiguity.
6. Published releases are monotonically numbered and immutable; restore creates a new draft.
7. The canonical editor and compiler preserve the full current side-panel rule catalog and lifecycle. Advanced nested query composition is additive, declarative, typed, explainable, and executable at every production consumer; no existing rule is removed to simplify the canonical model.
8. Neither the 4-of-5 exercise nor any blind agent interaction run is a completion gate. The self-guiding behaviors remain specified and receive source-aware automated production-boundary coverage; optional Windows usability evaluation remains outside this program.
